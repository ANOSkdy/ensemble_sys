## 1. 背景・目的

### 1.1 背景
- 40社規模×複数求人×頻繁な更新を、媒体管理画面で人手更新しており運用が飽和している。
- 顧客の主利用媒体が「Airワーク」であり、Airワークの一括入稿（Excelテンプレート / TSV）を軸に更新業務を標準化できる。
- 既存Ver.1では業務・データ・運用フローを整理済みであり、今回の更新では運用画面群の見た目とUI一貫性を固定し、実装ブレを防ぐ。

### 1.2 目的（Ver.2）
- 求人内容の「正（Single Source of Truth）」をNeonに集約し、入力画面で一元管理する。
- 会議メモを起点にAI（Gemini Flash 3.0）が改善案を提示し、承認後に一括入稿ファイルを自動生成して更新を高速化する。
- 掲載から14日超過した求人を「鮮度維持対象」として検知し、未掲載化→複製→新規掲載の運用を回せる状態にする。
- 画面/UIについては、本資料に定義する**デザインシステム・レイアウト・表示ルールを凍結**し、今後の基本デザイン判断の基準とする。

## 2. スコープ定義

### 2.1 対象（Ver.2で維持・明文化）
1) **求人データの正本DB**
- Airワーク一括入稿で用いる項目キーと整合する形で、求人情報を保持する（job_offer_id 等を含む）。

2) **AI改善提案（Gemini Flash 3.0）**
- 会議メモ＋現行求人＋制約を入力し、改善案（差分＋理由＋リスク）を構造化出力。

3) **Airワーク一括入稿ファイル生成**
- 生成形式は Excel（推奨）を基本とし、必要に応じてTSV（.txt）も扱う。
- 新規／更新は `job_offer_id` の有無で制御する。

4) **実行ログ・証跡**
- 生成ファイル、実行者、実行日時、対象求人、差分・ハッシュ等を保管する。

5) **鮮度維持（14日超過検知）**
- Cronで日次検知し、対象求人に対して未掲載化・新規掲載・同期のToDoを作成する。

6) **エラー取り込み支援**
- Airワーク側の入稿結果を取り込み、求人単位で原因表示・次回修正ToDo化を行う。

7) **デザイン仕様凍結**
- カラースキーム、イラスト要素、シャドウ、グラフ表現、レイアウト原則、画面区分け、アニメーション方針を本資料で固定する。

### 2.2 非対象
- Airワーク以外（en-gage、ハローワーク等）への一括更新
- Airワーク画面操作の完全自動化（RPA/ブラウザ自動操作）
- 有料広告設定、写真一括設定など、一括入稿で扱えない領域の完全自動化
- 大幅なブランド刷新や複数デザインテーマの併存対応

- 新規画面を追加する場合も、本資料のUI原則に従って拡張する。

## 4. デザインシステム基本方針

### 4.1 カラースキーム
| 区分 | 名称 | HEX | 用途 |
|---|---|---:|---|
| ベース | Base | `#F9F9F9` | 画面背景、空白領域、帳票背景 |
| プライマリ | Primary | `#4A90E2` | 主要CTA、選択状態、リンク、進行中表示 |
| セカンダリ | Secondary | `#50E3C2` | 補助アクション、成功系の強調、サブ指標 |
| アクセント-1 | Accent-1 | `#FFD166` | 注意、ハイライト、件数バッジ |
| アクセント-2 | Accent-2 | `#F25F5C` | エラー、危険操作、未解決警告 |
| アクセント-3 | Accent-3 | `#9D59EC` | AI関連導線、提案表示、分析カード |

### 4.2 配色ルール
- 背景はベースカラーを標準とし、白背景の多用を避けてまぶしさを抑える。
- 主要な確定操作はプライマリで統一する。
- AI提案や分析導線はアクセント-3を優先使用し、人手操作との差異を明示する。
- エラーやブロッカーはアクセント-2を使用する。
- 数値ダッシュボードのポジティブ変化はセカンダリ、注意喚起はアクセント-1を用いる。

### 4.3 イラスト要素
- 人物：頭身の低いシンプルなスタイル
- アイコン：2px線画 + 塗りつぶし
- 装飾：ドット、波線、パターン
- シャドウ：柔らかいドロップシャドウ

### 4.4 UIトーン
- B2B業務画面でありながら、堅すぎず親しみのある印象とする。
- 情報量は多いが、カード・余白・色分離で読みやすさを維持する。
- イラストや装飾は「補助」であり、入力・レビュー・実行を阻害しない。

## 5. レイアウト方針

### 5.1 画面構成
- セクションを明確に区分し、1画面内に複数責務がある場合もカードまたはパネル単位で分離する。
- 一覧 → 詳細 → 編集 → 実行 の遷移を直感的に追える構造とする。
- 主要操作は各セクションの右上、またはフッター固定領域に集約する。

### 5.2 プロセスフロー表現
- プロセスフローは**左から右**を第一選択とする。
- モバイルや縦長領域では**上から下**へ変換してよい。
- フロー内の状態は「未着手 / 進行中 / 完了 / 要対応」を色とラベルで明示する。

### 5.3 データ表示
- 円グラフ、棒グラフはフラットデザインとし、3D表現や強いグラデーションは用いない。
- 大量比較は棒グラフ、比率把握は円グラフを優先する。
- 数値カードと併用し、グラフ単体に意味を依存しない。

### 5.4 アニメーション
- 控えめなトランジションを標準とする。
- ローディングや差分展開には150〜250ms程度の短い遷移を適用する。
- 派手な移動、過剰な拡大縮小、常時点滅は禁止する。

## 6. 画面設計（基本方針）

### 6.1 必須画面
1) 認証画面
- ログイン
- ロール（Admin / Operator / ReadOnly）
- ブランド導入ビジュアルは簡素なイラストと最小限の説明に留める

2) CRM画面
- 顧客一覧/詳細
- 媒体アカウント（Airワーク）管理
- 顧客単位の状態が一目で判別できるカード/テーブル併用構造

3) プレビュー画面
- 差分表示（項目単位）
- AI提案の採用/却下/部分採用
- リスク警告表示
- before/after は左右比較を基本とする

4) 実行キュー（Runbook）画面
- 生成ファイル一覧（顧客別・日付別）
- 実行手順チェックリスト
- 実行証跡（スクショ/メモ/担当者/時刻）
- 実行ステータスはバッジと進捗ラインで表示する

5) インポート/同期画面
- 取り込み対象、結果、失敗要因をセクション分離
- 成功/失敗の件数をサマリーカードで先頭表示

6) Airワークコード/マスタ確認画面
- テーブル主体
- フィルタ・検索・最終更新日時を上部配置
- 変更有無を視覚的に識別できる配色を用いる

7) ToDo画面
- 優先度、期限、担当、状態を一目で見分けられる一覧構成
- 手動工程の抜け漏れを防ぐため、完了条件をカード内に明示する

### 6.2 共通UIパターン
- 一覧：カード＋テーブルのハイブリッド
- 詳細：上部サマリー、下部タブまたはセクション積層
- 編集：左に入力、右に補助情報/バリデーション/AI補助
- 承認：差分、理由、リスク、操作を縦の流れで配置
- 実行：対象一覧、検証結果、生成結果、証跡の4区分

## 7. コンポーネント基本方針

### 7.1 カード
- セクションの基本単位はカードとする。
- 角丸は中〜大きめ、柔らかいドロップシャドウを付与する。
- カード内は見出し、説明、本文、操作領域の4層を基本構造とする。

### 7.2 ボタン
- Primary：プライマリ色背景、白文字
- Secondary：白または淡色背景、境界線あり
- Danger：アクセント-2
- AI操作：アクセント-3系統
- 同一画面で主ボタンは1〜2種類までに絞る

### 7.3 バッジ/ステータス
- draft：中立色
- in_review / executing：プライマリ
- approved / done：セカンダリ
- warning / pending：アクセント-1
- failed / blocked：アクセント-2
- ai / analysis：アクセント-3

### 7.4 フォーム
- ラベルは常時表示
- 必須/任意は明示
- エラーはフィールド直下
- 依存項目は段階表示し、不要項目は隠すか無効化する

### 7.5 アイコン/装飾
- 操作アイコンは2px線画 + 塗りの統一ルールに従う
- ドットや波線パターンは空き領域やガイド要素に限定利用する
- 業務データ本体の可読性を損ねる装飾は禁止する

## 8. 業務仕様（継承）

### 8.1 一括入稿の制約
- 一括入稿では「入力不可」の項目が存在するため、アプリ側で参照専用として扱う。
- 未掲載化、掲載、再掲載など画面操作が必要な工程はToDoで運用する。

### 8.2 新規／更新の分岐
- `job_offer_id` が空欄：新規作成
- `job_offer_id` が存在：更新
- 生成時はこの分岐を厳密に守る。

### 8.3 コード厳密性と入力条件
- コードを含む値は文字列として扱う。
- 条件依存フィールドの矛盾は出力前に排除し、警告表示する。

## 11. セキュリティ設計
- RBAC（Admin/Operator/ReadOnly）
- DB接続・Blob・GeminiキーはVercel環境変数のみ
- repoには`.env.example`のみ
- 媒体ログイン情報は原則平文保管しない
- 監査ログで承認・生成・完了操作を追跡する

---

## 12. 非機能要件（UI/UX）
- 初見ユーザーでも「今どの工程か」が把握できること
- エラーは赤一色に頼らず、文言・アイコン・補助説明で伝えること
- 操作頻度の高い画面では、1スクリーン目に重要情報を集約すること
- AI提案と人手確定操作を視覚的に区別できること
- 一覧画面は顧客数・求人数が増えても判読性を保てること

1) 既存Ver.1の業務要件を満たした上で、デザイン仕様が全主要画面に一貫適用されている

4) グラフ・進捗・フロー表現がフラットデザインで統一されている
